Vulhub学习（一）

发表于 2020-07-15

zabbix latest.php SQL注入漏洞（CVE-2016-10134）

启动Docker

  • 因为下载速度太慢，要把docker镜像源换成国内的
  • 编辑 /etc/docker/daemon.json，粘贴以下内容
  • 重启docker
  • 发现报了 Job for docker.service failed 的错误
  • 参考这篇文章：Job for docker.service failed 修改docker镜像源遇到的坑
  • 将 daemon.json 改名为 daemon.conf，就可以了
  • 然后执行以下命令启动

配置虚拟机

  • 我使用的虚拟机是VirtualBox，连接方式是网络地址转换(NAT)
  • 为了使宿主机能访问虚拟机中的服务，需要配置端口转发
  • 管理 → 全局设定 → 网络，添加以下网络

  • 选择虚拟机 → 设置 → 网络 → 高级 → 端口转发

  • 宿主机就可以通过 http://127.0.0.1:8080 访问服务了

测试漏洞

  • 访问 http://127.0.0.1:8080/jsrpc.php?type=0&mode=1&method=screen.get&profileIdx=web.item.graph&resourcetype=17&profileIdx2=updatexml(0,concat(0xa,user()),0)

分析源码

下载旧版本源码

  • 在zabbix官网放出来的源码都是已经修复好漏洞的版本，找了很久也没找到存在漏洞的版本

  • 后来发现它的下载链接是这样的 https://cdn.zabbix.com/zabbix/sources/oldstable/2.2/zabbix-2.2.23.tar.gz
  • 于是访问上级目录 https://cdn.zabbix.com/zabbix/sources/oldstable/2.2/，发现了老版本

  • 下载 zabbix-2.2.10

代码审计

jsrpc.php

  • 找到 jsrpc.php 的文件 zabbix-2.2.10\frontends\php\jsrpc.php
  • 可以看出请求的参数绑定到了 $data 变量中
  • 其中根据 method 参数执行不同的操作
  • 根据POC中的 method=screen.get，找到 case 'screen.get':
  • profileIdx2 参数被赋值到了 $option 变量，再通过 CScreenBuilder::getScreen($options); 处理 $option 变量

CScreenBuilder.php

  • 跟踪到 zabbix-2.2.10\frontends\php\include\classes\screens\CScreenBuilder.php
  • 在它的构造函数中，将 profileIdx2 参数传到了 CScreenBase::calculateTime()

CScreenBase.php

  • 跟踪到 zabbix-2.2.10\frontends\php\include\classes\screens\CScreenBase.php 的 calculateTime() 方法
  • 根据if判断，profileIdx2 传入了 CProfile::update()

profiles.inc.php

  • 跟踪到 zabbix-2.2.10\frontends\php\include\profiles.inc.php 的 update() 方法
  • 发现 profileIdx2 被赋值到了 self::$insert[$idx][$idx2]
  • 但是至今还没有执行数据库操作

page_footer.php

  • 回到 jsrpc.php，文件的最后调用了 page_footer.php
  • 跟踪到 zabbix-2.2.10\frontends\php\include\page_footer.php
  • 其中46行调用了 CProfile::flush();

profiles.inc.php

  • 跟踪到 zabbix-2.2.10\frontends\php\include\profiles.inc.php 的 flush() 方法
  • 其中将 profileIdx2 参数传入了 self::insertDB()
  • 跟踪到 insetDB() 方法
  • 发现 profileIdx2 参数没有经过处理就拼接到了SQL语句中，SQL注入就在这里产生
